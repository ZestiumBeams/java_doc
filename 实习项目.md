# 实习项目

##### **参与 ERP 系统可复用的 Excel 读写组件的开发**

基于 Hutool POI 实现，通过泛型结合自定义注解实现 Excel 与 Java Bean 的自动映射，同时采用模板方法和策略模式解耦读取与写入逻辑，并支持动态字段解析与多种导出模式。提高了 Excel 导入导出的复用性和扩展性、覆盖表单、商品、库位等批量维护场景。

###### 业务需求

这个组件把 OneFlow 所有业务域的 Excel导入导出需求抽象成一个 starter，让业务开发只需要关注列映射定义，而把 Excel读写、样式、校验、容错统一交给组件处理。在系统中，原来的是已经封装好的有，只不过是针对动态excel表头的，需要在业务层去处理表头信息。

###### 介绍一下这个组件？

1️⃣ 这个组件的作用主要就是封装 Hutool POI 的excel读写，将读写逻辑写到组件中的redisTemplate类中，对外暴漏redisTemplate。通过自定义注解绑定业务类中属性和对应excel列的关系，继承AbstractSheetReader抽象类，重写read\(\)方法实现通过注解的方式解析和动态字段解析。

2️⃣ 动态excel模板读写是前端控制的，是运行时生效，不能写死在注解里面。需要在业务中处理，封装到一个通用的字段和excel列关系的类中，再作为参数传到通用组件库中。

3️⃣ ExcelReader\.read（）是一个模板方法，接受一个AbstractSheetReader抽象类的子类。

它的解析流程是这样的：先用 ExcelReader 把 Excel 文件解析成 Sheet，然后框架会回调我们实现的 `read(Sheet)` 方法。进入方法后，先读取表头那一行，通过 `rowToHeader` 给每个字段建立一个“字段 → 列索引”的映射，为了效率表头的映射这一步只做一次。之后逐行读取数据，每一行根据这个映射直接取对应列的值，再通过反射把值填充到 Java Bean 里，最终组成 List 返回。

本质上就是**先做结构映射，再做数据填充，用一次表头解析换后面所有行的高效转换**。

###### 问题？

---

✅ Q1：为什么不用 Apache POI / Hutool 直接写？

👉 标准答案：

> POI / Hutool 解决的是“如何读写 Excel 文件”，但没有解决“业务映射”的问题。
> 在 ERP 场景中，每个导入都需要手写字段解析、类型转换、校验逻辑，代码重复且难维护。
> 我这个组件的核心价值是：**把 Excel 结构解析和业务映射解耦**，通过注解 \+ 反射实现通用化，提升复用性和开发效率。
> 
> 

---

✅ Q2：模板方法模式体现在哪？

👉 标准答案：

> 模板方法在 `AbstractSheetReader.read()` 中定义了解析流程：
> 
> 读取Sheet → 解析表头 → 遍历行 → 转换对象
> 
> 具体实现由子类（BeanSheetReader / DynamicSheetReader）完成，比如：
> 
> rowToHeader
> 
> rowToBean
> 
> 这样流程固定，细节可扩展。
> 
> 

---

✅ Q3：为什么要 rowToHeader？

👉 标准答案：

> 为了提前建立字段 → 列索引的映射关系。
> 如果不这样做，每一行都要通过 `indexOf` 查找列，会从 O\(n\*m\) 变成 O\(n²\)，性能会明显下降。
> 同时也让解析逻辑更清晰，职责分离。
> 
>

🔥 第二轮：原理

---

✅ Q4：为什么 read\(sheetReader\) 会调到你的 read？

👉 标准答案：

> 这是典型的回调机制 \+ 多态。
> ExcelReader 在内部会调用：
> 
> handler\.read\(sheet\)
> 
> 由于传入的是 BeanSheetReader 实例，根据 Java 动态绑定机制，会执行我们重写的 `read(Sheet)` 方法。
> 如果换成 DynamicSheetReader，就会执行对应实现。
> 
> 

---

✅ Q5：反射性能怎么优化？

👉 标准答案（一定要具体）：

> 可以从几个方面优化：
> 
> **缓存 Field 信息**（避免重复反射扫描）
> 
> **提前构建字段映射（已做）**
> 
> 使用 **MethodHandle / LambdaMetaFactory** 替代反射
> 
> 批量导入场景可减少反射调用次数（比如批量 setter）
> 
> 但在 Excel 导入场景中，IO 是主要瓶颈，反射影响相对可控。
> 
> 

---

✅ Q6：row\.indexOf 是 O\(n\)，怎么优化？

👉 标准答案：

> 可以在解析表头时先构建一个：
> 
> Map\<title, index\>
> 
> 这样查找复杂度从 O\(n\) 降到 O\(1\)，在列很多时效果明显。
> 
>

⚔️ 第三轮：设计

---

✅ Q7：为什么分 Bean / Dynamic？

👉 标准答案：

> 两者本质是两种不同的映射方式：
> 
> Bean：编译时注解驱动（结构固定）
> 
> Dynamic：运行期配置驱动（结构不固定）
> 
> 如果强行合并，会导致：
> 
> 逻辑复杂
> 
> 判断分支过多
> 
> 拆分后更清晰，也符合单一职责原则。
> 
> 

---

✅ Q8：支持100万行怎么办？

👉 标准答案（重点🔥）：

> 当前实现是基于内存的，不适合大文件。
> 可以优化为：
> 
> 使用 **SAX（事件流）模式解析（如 POI SAX）**
> 
> 分批处理（分页入库）
> 
> 边读边处理，避免 List 全量加载
> 
> 配合消息队列异步处理（如 Apache RocketMQ）
> 
> 核心思想：**流式处理 \+ 降低内存占用**
> 
> 

---

✅ Q9：列顺序变了会挂吗？

👉 标准答案：

> 如果使用 columIndex，会受影响；
> 如果使用 columTitle，则不会，因为是按标题匹配。
> 
> 所以在实际使用中：
> 
> 推荐用标题匹配提升鲁棒性
> 
> index 适用于固定模板
> 
>

✅ Q10：数据类型错误怎么办？

>

>

>

>

##### **参与单据编号功能开发**

支持按日期、员工、部门、门店等维度动态生成业务单号。基于 Redis 实现编号预占用，使用 zset 数据结构维护编号回收池、实现提交确认及取消报废场景下的流水号回收与重新分配。

###### 业务背景，为什么要回收利用单据编号？

系统中创建单据号是基于 Redis 的原子自增命令** INCR**。在创建订单时，会生成单据编号，但是如果前端并没有提交单据，直接取消或者保存草稿后又作废了。那再创建另一个单据时生成的单据号就是断的，不连续的。

根据业务要求，ERP（尤其财务相关）里是不允许段号的。中国税务/财务审计要求单据编号连续不可断（《会计法》相关条款）。仓库按单拣货、财务对账都依赖编号连续性等。

这样就会涉及到生成编号的复用和回收。

###### 怎么实现？

![Image](https://internal-api-drive-stream.feishu.cn/space/api/box/stream/download/authcode/?code=ZDMxZWEyNjFmMGE4MDQ1YTQ0N2ZlNmYyOGQ4ZGI0NzJfZTg0NjNjMjMzZTU4ZThmZTViMTBhMzljNzgyOGVmMGNfSUQ6NzYzNTA5NjM0MDI5NjQ1MzA3MV8xNzgxMDU1NjA3OjE3ODExNDIwMDdfVjM)

1️⃣ 在创建单据时，通过 redis incrby 生成流水号（根据业务需求，有些是每天从 0 开始生成，有些是永久有效），并同时用 uuid 生成一个业务 id 标识绑定流水号和业务单据的关系用来后续提交时从 redis 中删除或者回收时的幂等性。

2️⃣ 使用 zset 维护一个编号的回收池，score 分数和流水号一致。创建单据时，如果单据提交，根据 vchCode 从 zset 回收池中删除对应的流水号；如果取消订单或者废弃时，需要将流水号放到回收池 score 分数和流水号一致。这样下次使用流水号时优先从 zset 里面根据分数获取最小的也就是最先被创建的流水号。、

###### 为什么要 vchCode？

vchCode 就是一个一次性令牌，用于标记"这次编号生成操作"，防止重复提交、重复回收，并在门店切换时识别编号的归属地。 每次生成编号都会产生一个新的 vchCode，提交或回收后这个 vchCode 就失效了（从 Redis 删除或标记），24 小时自动过期兜底。

消息推送组件                                                                                           



背景："项目中原有的消息推送比较零散，各业务模块各自实现通知逻辑，没有统一的消息管理。同时需要支持

App、小程序、Web 三端推送，并且消息要持久化，方便用户后续查阅。"                                     



难点：

- 腾讯 IM 的接入有一定复杂度（UserSig 签名、离线推送配置、多端兼容）

- 消息与 21 种业务单据的关联关系管理

- 消息的已读/未读、忽略、删除等状态维护

    

    方案："我把它封装成了一个独立的 Spring Boot Starter，核心做了三件事：

1. 封装了腾讯 IM 的 SDK 调用，包括消息推送、用户账号同步、离线推送（Android/iOS/鸿蒙三端）

2. 设计了统一的消息数据模型，把销售、采购、调拨等所有业务场景的消息统一入库，用枚举来区分消息类型

3. 对上层业务屏蔽推送细节，业务方只需要调用一个接口就能完成消息持久化\+多端推送

    

    同时基于这套体系做了预警提醒功能——库存超限/不足、批次到期、逾期账单，都会自动生成消息推送给对应用户。"

    

    成果："收口了全项目的消息推送入口，新增业务消息只需要加一个枚举值，无需重复开发推送逻辑。"

仓储库位管理



背景："进销存系统中仓库管理是核心模块，但之前的货位管理比较简单，无法满足精细化管理需求。比如有的客户只

需要简单的库位编号，有的客户需要按货区→通道→货架→层→货位号五级维度来管理。"



难点：

- 库位编码规则可配置，不同公司不同维度

- 库位与 SKU 的绑定关系需要支持批量操作、导入导出

- 绑定/解绑后库位状态的自动流转

    

    方案："我设计了简单库位和标准库位两种模式，标准库位支持三级维度配置（管理维度 \+ 各段位长）。通过

    StoreLocationConfig 表按公司维度存储配置，首次查询自动填充默认值。

    

    核心是 StoreLocationGoodsRelation 这个绑定关系表，围绕它实现了：

- 多条件组合查询（仓库、SKU、属性、条码、绑定状态等过滤）

- 带事务的批量绑定/解绑，自动更新库位状态（绑定→使用中，解绑→闲置）

- Excel 导入导出，异步处理大批量数据，WebSocket 实时推送导入结果"

    

    成果："支持了不同客户的个性化库位管理需求，从简单的编号到五级精细化货位都能灵活配置。批量导入解绑时通过

    事务保证了数据一致性。"
