## 项目中的幂等性

**保证幂等性的方案**

**1️⃣ 分布式锁**

用 接口路径 \+ 当前用户ID \+ 参数MD5 生成唯一锁 key，然后抢 Redisson 分布式锁；抢不到就认为是重复提交（不使用）。

2️⃣ **请求token**

项目中的接口的幂等性使用这种方案。先通过 /token 申请一个幂等 token，请求时带上它；服务端校验时直接删 token，删成功说明第一次用，删失败说明已用过或失效。**这种方案依赖于前端**。

**3️⃣ 去重表**

去重表是指在使用 Redis 或者 MySQL 作为存储时，为了实现幂等性而用于记录已经处理过的请求或操作，以防止重复执行。大部分场景下，大家会使用 Redis 作为去重组件实现。

项目中通过写一个**自定义注解**，然后用Spring AOP 写这个同一处理的**@Around**类型的切面类，对接口方法或者MQ消费者的**onMessage\(\)**方法注解上的标识去做幂等性判断。

### 一、接口幂等

**接口幂等性是指一个接口支持幂等性，同一个请求那么无论调用多少次，产生的结果是一样的。**防止接口重复提交，或者因为网络波动发出多次请求。

幂等避免的是是重复请求（同一个请求发了两次）而不是新的业务请求。

**支付接口的幂等**

使用分布式锁的的方式，幂等键使用订单号作为锁标识。不过这里锁释放后还是会有其他的重复创建订单，本质上是并发防重，不是永久去重。但是这里做了一个缓存存储这个订单号对应的支付结果，能保证一定时间内不会反复请求创建支付。

**@Idempotent**：防并发重复创建                                                                       

**distributedCache**：防短时间内重复请求反复创建支付结果

### 二、MQ幂等

**消息队列幂等是指保障消息队列客户端对相同的消息仅消费一次。**

在 MQ 场景下，采用基于 Redis 的消费状态机来实现幂等。通过 Lua 脚本保证原子性，将消费状态分为 CONSUMING 和 CONSUMED。当消息第一次消费时写入 CONSUMING，表示正在处理；其他消费者来时，如果再次消费发现是 CONSUMING，说明结果未确定，需要抛异常让 MQ 重试；如果是 CONSUMED，则说明已经处理成功，直接忽略。业务执行成功后更新为 CONSUMED，失败则删除 key，**从而保证消息不会丢失且不会被重复处理**。

### 一、使用**责任链模式**重构请求参数检验，如：查询购票、购买车票下单以及支付结果回调

###### 责任链模式是什么？为什么要使用责任链模式？不使用会怎样？

1️⃣ 责任链模式是说一个请求，会经过多个处理器去执行，上一个处理器执行完把结果传给下一个，以此类推形成一个链式的处理模式，链条上面每个处理器各司其职。

2️⃣ 项目使用责任链模式并不是套用设计模式，结合这个项目，像购票下单前的大量参数校验这类处理校验天然就是一组有**顺序**、**可扩展**、**可失败即中断**的规则。有些校验比较轻比如校验参数是否完整可以放到前面不合法就直接抛出异常，有些校验需要查缓存/数据库，比如购票时查询用户或者车次是否存在放到后面校验。对后续的校验进行扩展时，只需要增加责任链中的处理器（项目中就是实现对应的** AbstractChainHandle**）**，**并标注业务标识和 Order 值控制在责任链中的执行顺序，不用改任何代码。符合开闭原则。

3️⃣ 如果不使用完全没问题，只是在业务层里面会写一堆的 if\-else，顺序容易乱，比如还没做非空校验就去查库。以后修改时还要再原有的类和方法中做修改，不符合开闭原则。

###### 项目中的具体实现？

1️⃣ 定义抽象处理器接口** AbstractChainHandler**，在接口内部定义处理器的方法和责任链的业务标识，并且这个接口继承 Spring **Ordered， **作为同一个责任链中执行优先级的标识。

```Java
public interface AbstractChainHandler<T> extends Ordered {

    */***
*     * 执行单个责任链逻辑*
*     **
*     * @param requestParam 责任链执行入参*
*     */*
*    *void handler(T requestParam);

    */***
*     * @return 责任链组件标识*
*     */*
*    *String mark();
}
```

2️⃣ 根据业务去实现单个的处理器接口，并定义自己的处理逻辑，执行顺序和业务标识。

3️⃣ 创建一个** AbstractChainContext **容器，并实现** ****CommandLineRunner**** **接口，重写 **run **方法，在 run 方法中通过**反射**获取所有定义的处理器接口的实现类，按照业务标识分组，按照 Ordered 值排序。以业务标识为 key， 同一个责任链下的处理器组成的 List 为 value 的这样一个 Map 当整个项目中的责任链上下文容器 （封装到一个启动器中）。

```Java
public final class AbstractChainContext<T> implements CommandLineRunner {

    // 使用map模拟存放多条责任链的容器，支持多个业务链
    private final Map<String, List<AbstractChainHandler>> abstractChainHandlerContainer = new HashMap<>();

    */***
*     * 责任链组件执行*
*     **
*     * @param mark         责任链组件标识*
*     * @param requestParam 请求参数*
*     */*
*    *public void handler(String mark, T requestParam) {
        // 根据mark 获取责任链组件
        List<AbstractChainHandler> abstractChainHandlers = abstractChainHandlerContainer.get(mark);

        // 责任链组件不存在, 抛异常
        if (CollectionUtils.*isEmpty*(abstractChainHandlers)) {
            throw new RuntimeException(String.*format*("[%s] Chain of Responsibility ID is undefined.", mark));
        }

        // 对责任链List中的顺序进行执行
        abstractChainHandlers.forEach(each -> each.handler(requestParam));
    }

    */***
*     * CommandLineRunner中的启动方法，Spring Boot启动. Spring容器完全初始化后执行 run()方法*
*     * @param args*
*     * @throws Exception*
*     */*
*    *@Override
    public void run(String... args) throws Exception {
        // 获取所有AbstractChainHandler
        Map<String, AbstractChainHandler> chainFilterMap = ApplicationContextHolder
                .*getBeansOfType*(AbstractChainHandler.class);

        // 根据责任链中单个组件标识进行分组
        chainFilterMap.forEach((beanName, bean) -> {
            List<AbstractChainHandler> abstractChainHandlers = abstractChainHandlerContainer.get(bean.mark());

            // 可能为空（第一个）
            if (CollectionUtils.*isEmpty*(abstractChainHandlers)) {
                abstractChainHandlers = new ArrayList();
            }

            abstractChainHandlers.add(bean);

            // 按照继承得到Ordered接口的getOrder方法返回值进行排序, // *TODO 这里有个小问题，每次循环都会重新排序*
*            *List<AbstractChainHandler> actualAbstractChainHandlers = abstractChainHandlers.stream()
                    // 按照继承得到Ordered接口的getOrder方法返回值进行排序， 默认升序
                    // 执行顺序就是从小到大
                    .sorted(Comparator.*comparing*(Ordered::getOrder))
                    .collect(Collectors.*toList*());

            // 按照mark添加到map容器中
            abstractChainHandlerContainer.put(bean.mark(), actualAbstractChainHandlers);
        });
    }
}
```

4️⃣ 在使用时，只需注入上下文容器的依赖，调用 hander 方法，传入责任链的标识也就是 Map 中的 key，在 hander 方法里面遍历 list 中的处理器顺序执行。

###### Rocket MQ 延迟消息底层原理？

当消费者发送延时消息时，会被 broker 转发到**延迟 topic **的队列中，然后 broker 后台有一个线程** ScheduleMessageService **会一直对每个队列对应的定时任务 delay 不停的检查是否到设定的时间（轮询扫描），等到检查到达延迟时间转发到原始的 Topic 中并去掉 delay 属性，就能继续被各自的消费者消费了。

SpEL 表达式：使用接收到消息的 keys（也就是订单号）拼接下划线加接受整个消息体的 hash 值。作为幂等键，然后在自定义注解标注的方法作为切点，定义一个切面类，以 Redis 做为去重表，存放这个幂等标识。

###### 为什么延迟关闭订单发送端在订单服务，消费延迟关闭订单消息却是在购票服务？

会造成**微服务的远程调用循环**。

列车购票是从购票服务发起的，通过**购票服务**分配座位、修改座位状态以及创建订单，已知购票服务调用了订单服务。而延迟关闭订单中，涉及到订单状态的变更和列车座位的变更等，这样就涉及到两个服务的调用。如果消息消费在订单服务，就涉及到远程循环依赖，这对于微服务涉及中是需要尽可能避免的。所以，最终将消费延迟关闭消息放在了购票服务。

### 三、封装缓存组件库避免注册用户时，用户名全局唯一带来的缓存穿透，减轻数据库访问压力

###### 如何对用户进行分库分表的

设计到分库分表，分片键是一个最重要的，查询时如果没有带上分片键，会触发**广播查询**，对所有库所有表进行查询。

```YAML
rules:
  - !SHARDING
    tables:
      t_user:
        actualDataNodes: ds_${0..1}.t_user_${0..31}
        databaseStrategy:
          standard:
            shardingColumn: username
            shardingAlgorithmName: user_database_hash_mod
        tableStrategy:
          standard:
            shardingColumn: username
            shardingAlgorithmName: user_table_hash_mod
            
            
    *# 配置分片算法*
shardingAlgorithms:
  user_database_hash_mod:
    type: CLASS_BASED
    props:
      sharding-count: 32
      table-sharding-count: 16
      strategy: standard
      algorithmClassName: org.opengoofy.index12306.framework.starter.database.algorithm.sharding.CustomDbHashModShardingAlgorithm
```

这里分库和分表都使用**用户名**为分片键，**分片规则**使用哈希取模（user\_database\_hash\_mod）。

比如，用户名是：`username = "zhangsan"`，计算

`hash(username) % 2 = 0  `

`hash(username) % 16 = 8  `

可以确定在 ds\_0\.t\_user8，分库和分表使用同一个分片键的好处：一次 hash 就能确定库 \+ 表，不需要广播。

###### 业务场景

注册用户时要进行用户名是否存在的判断，每次输入用户名都会去查询数据库尝试对比是否合法。当大量的请求打到数据库可能会造成**缓存穿透**，在高并发的情况下，可能有大量的新用户同时注册，输入的用户名极有可能都不存在于数据库中。这将导致大量的缓存不存在，都去查询数据库，造成数据库压力剧增。

###### 解决方法

![Image](https://internal-api-drive-stream.feishu.cn/space/api/box/stream/download/authcode/?code=MmQ5YmRmMWNkNjZhZjdhNjY1NzI2NTVkZjQ2MWYxM2ZfZDhhNjNhMmRkYmU5ZDc5ZTI5MmNlNmNhZTE2YTYxYWNfSUQ6NzYyODk5MjMxNjE2NTk2NjgwNV8xNzgxMDU1NTkyOjE3ODExNDE5OTJfVjM)

解决方案是**布隆过滤器** \+** Redis set 结构：**

1️⃣ 当注册请求向数据库查询是否该名字重复时，先去布隆过滤器中查找。

2️⃣ 如果布隆过滤器中没有，就直接是认为名字合法，放行并加入布隆过滤器。

3️⃣ 如果布隆过滤器中有（这个可能没有，因为有哈希冲突），有的话也可能是已经注销的用户的名字，要去再查缓存，redis 中用一个 set 去存放每次注销的用户的名字。

4️⃣ 如果布隆过滤器中有且 redis 中有，说明这个名字是可以复用的。

使用 redis set 存放已注销的用户的用户名，是因为布隆过滤器里面的数据是不能删除的。

###### 为什么不使用其他的缓存穿透的解决方案

**缓存穿透：**是说请求访问一个根本不存在的值时，会每次都到达数据库并且不会重建缓存，那么当有大量这样的请求到来时，数据库的压力骤增。

使用**缓存空值**的做法，在这里的弊端是，如果每次输入一个名字缓存中都不存在，缓存 null，TTL 设为 60 秒。那么在这 60 秒内，无论该用户是否使用这个名字，其他的用户都不能使用

使用**分布式锁**的方案，每次保证只有一个线程去访问数据库重建缓存在高并发的条件下可能会导致请求超时。

###### 介绍一下Binlog 

BinLog是数据库操作变更日志，**MySQL会把增删改的操作记录到日志文件中**。BinLog会记录哪一张张表，执行了什么操作，然后那个数据变化。BinLog的作用是**主从复制**，**数据恢复**和**同步**。

###### 常见的保证数据一致性问题的方案？

1️⃣ 先修改数据库，再删除缓存

在极端情况下会造成缓存与数据库数据不一致的情况：

读线程A，写线程B同时访问一个数据。此时缓存中没有这个数据，读线程回去读数据库，在读线程A读完数据库到重新写入缓存这段时间，写线程B完成了修改并更新缓存，线程A将刚才查到的旧值写入缓存，此时数据库和缓存的数据是不一致的。

如果删除失败了有什么解决方案吗？

将失败的消息加入消息队列（MQ），由消费者不断重试直到删除成功或者最终丢到死信队列中人工解决。

2️⃣ 延时双删

先删除缓存，再更新数据库，延时几百毫秒再次删除缓存（**防止更新数据库期间有旧数据重新进入缓存**）。

3️⃣ canal 监听数据库配合MQ异步更新缓存

解决这个问题的核心不在于杜绝所有冲突，而是保证最终一致性，利用**订阅 Binlog 异步同步缓存数据**来最大程度降低不一致的时间窗口。

### 五、通过订单号和用户信息复合分片算法完成订单数据分库分表，支持订单号和用户查询维度

###### 为什么要使用复合分片算法？不使用行不行？

如果单纯的使用userId分表，在查询订单的业务中，有些接口中是没办法直接传userID的。因为订单中的userId是下单人的userId，可是可能车票是其他人帮买的。这种情况如果你要查询本人车票订单，只能使用订单号。

因此，订单表（包括订单详情表）分片时要综合考虑订单号和用户id。

如果不使用复合分片的话，就要在数据库创建一个中间路由表，idCard 和订单号的关系表。

###### 订单表的复合分片算法

从业务出发，订单的查询涉及用户名查询和根据订单号精准查询订单。这里的分片键使用 **用户 id** \+ **订单 id **的复合分片策略。分片时使用用户 id 的后六位分库分表，生成订单号时在订单号后面携带用户 id 的后六位。

继承复合分片算法类** ComplexKeysShardingAlgorithm **再重写 **doShareing** 分片算法：

```TypeScript
*/***
* * 根据分片键获取数据库分片， 重写doSharing()计算出当前 SQL 应该路由到哪个数据库（ds_0、ds_1...）*
* * @param availableTargetNames 可用的数据库表名：Collection availableTargetNames = ["ds_0", "ds_1", "ds_2", "ds_3"];*
* * @param shardingValue 分片键*
* * @return 数据库表名字*
* */*
@Override
public Collection<String> doSharding(Collection availableTargetNames, ComplexKeysShardingValue shardingValue) {
    // *NOTE 从 ComplexKeysShardingValue 中获取“分片字段 → 分片值”的映射关系*
*    *// 比如：user_id -> [123456789]
    // 或者：order_sn -> ["20240101123456"]
    Map<String, Collection<Comparable<?>>> columnNameAndShardingValuesMap = shardingValue.getColumnNameAndShardingValuesMap();
    Collection<String> result = new LinkedHashSet<>(availableTargetNames.size());
    if (CollUtil.*isNotEmpty*(columnNameAndShardingValuesMap)) {
        String userId = "user_id";
        Collection<Comparable<?>> customerUserIdCollection = columnNameAndShardingValuesMap.get(userId);

        //*NOTE  如果分片建中存在用户 ID, 直接根据用户 ID 后六位作为分片键*
*        *if (CollUtil.*isNotEmpty*(customerUserIdCollection)) {
            String dbSuffix;
            Comparable<?> comparable = customerUserIdCollection.stream().findFirst().get();
            if (comparable instanceof String) {
                String actualUserId = comparable.toString();
                // *NOTE 对sharingCount（32）取模，再除以一个库中表的数量（16） 得到 0-1 直间的数；锁定数据库的编号*
*                *dbSuffix = String.*valueOf*(hashShardingValue(actualUserId.substring(Math.*max*(actualUserId.length() - 6, 0))) % shardingCount / tableShardingCount);
            } else {
                dbSuffix = String.*valueOf*(hashShardingValue((Long) comparable % 1000000) % shardingCount / tableShardingCount);
            }
            result.add("ds_" + dbSuffix);
        } else {
            // *NOTE 如果分片键中不存在用户 ID, 则根据订单号后六位作为分片键*
*            *String orderSn = "order_sn";
            String dbSuffix;
            Collection<Comparable<?>> orderSnCollection = columnNameAndShardingValuesMap.get(orderSn);
            Comparable<?> comparable = orderSnCollection.stream().findFirst().get();
            if (comparable instanceof String) {
                String actualOrderSn = comparable.toString();
                dbSuffix = String.*valueOf*(hashShardingValue(actualOrderSn.substring(Math.*max*(actualOrderSn.length() - 6, 0))) % shardingCount / tableShardingCount);
            } else {
                dbSuffix = String.*valueOf*(hashShardingValue((Long) comparable % 1000000) % shardingCount / tableShardingCount);
            }
            result.add("ds_" + dbSuffix);
        }
    }

    return result;
}
```

表的分片算法几乎一模一样，只不过是 **hashShardingValue\(后六位\) % shardingCount 。**

### 六、12306好像没有对分布式事务进行处理，这个问题是怎么解决的？

除非强一致性的业务会用Seata分布式事务 一般都会采用最终一致性的解决方案 比如本地消息表\+定时任务或者RocketMQ的事务消息  12306目前不咋需要分布式事务去解决 下单失败会抛出异常 两个服务都会回滚。

这个项目里，只涉及对票和对订单的表的改动。而创建订单是订票服务的远程调用，远程调用是有返回值，那么异常可以让订票服务回滚。

而真的存在分布式的问题的是那种比如有 订单创建、扣减库存、删购物车，这种模型。订单创建后远程调用其他两个服务，相当于一个父两个子，那么两个子之间是没法相互感知的，所以会存在分布式事务的问题。
